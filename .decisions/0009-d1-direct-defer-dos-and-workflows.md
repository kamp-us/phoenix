# 0009 — D1-direct; defer DOs, projection, and workflows

Supersedes [0003](0003-pasaport-singleton-do.md), [0005](0005-product-dos-shard-by-coordination-atom.md), [0006](0006-product-dos-extend-cloudflare-agent.md), [0008](0008-mutations-as-workflows.md).

## Context

ADRs 0003–0008 built up, in seven days, a stack of: Pasaport singleton DO,
sharded product DOs (`SozlukTerm`, `PanoPost`) extending Cloudflare's `Agent`
base, an outbox-in-DO durability pattern, a `PhoenixProjection` workflow
materialising D1 read views, and finally a workflow-as-coordinator mutation
pattern with 1:1 GraphQL-mutation ↔ workflow-class naming. The most recent
ADR (0008, 2026-05-10) superseded 0007 specifically because cross-DO
mutations needed a coordinator. The 2026-05-15 architecture audit surfaced
seven candidates; most of them are friction generated by the multi-layer
write path itself (outbox vocabulary still live, tests asserting against
DO-internal sqlite, vote semantics distributed across DO + projection +
resolver + Relay, error mapping duplicated per feature).

Phoenix is pre-production. Zero users, zero live data. The cross-DO
mutations the 0008 ADR was designed around (invite-spending, tree-pruning
ban) are part of the Künye achievement system — a feature that hasn't been
built yet and whose v1 scope is rep-only with Lobsters-shaped self-regulation.
None of the scaling pressures the DO + projection + workflow stack solves
exist today.

The cost we're paying:
- ~2400 lines of DO + outbox + workflow + reconciliation code in
  `apps/web/worker/features/{sozluk,pano,pasaport}/` and `worker/view/`
- 19 integration tests coupled to `runInDurableObject`, outbox row counts,
  and projection convergence loops
- Two retired ADRs (0004, 0007) plus a third (0008) implemented only in
  scaffolding
- A mental model that new contributors have to load before they can change
  anything

The simpler shape that delivers the same product surface today:
**resolvers write D1 directly**.

## Decision

### One D1, no DOs, no workflows, no projection layer

`PHOENIX_DB` is the single authoritative store. It is not a "view" or a
"projection" of anything — it holds the canonical state for every product
domain (sozluk, pano, pasaport/auth).

- **Resolvers read and write D1 directly** via drizzle. The Effect
  `resolver()` wrapper continues to host runtime + auth context.
- **No Durable Objects.** Delete `SozlukTerm`, `PanoPost`, `Pasaport` DO
  classes and their `drizzle-orm/durable-sqlite` schemas. Remove all
  `*_OBJECT` bindings from `wrangler.jsonc` with the appropriate
  `delete_classes` migration entry.
- **No Workflows.** Delete `PhoenixProjection` and any workflow
  scaffolding. Remove the `workflows` array from `wrangler.jsonc`.
- **No outbox, no `flushOutbox`, no `reconcileOutbox`, no
  `processed_event` table, no `last_event_id` ordering guard.** D1 is the
  one writer; ordering is whatever drizzle queues into it.

### Pasaport becomes a module backed by D1

The auth tables (`user`, `session`, `account`, `verification`, `apikey`)
move from Pasaport's DO-internal sqlite to `PHOENIX_DB`. `better-auth` is
re-instantiated against a D1-backed drizzle adapter. The `Pasaport` class
disappears; the auth handler is wired directly into the Hono router. RPC
call sites (`env.PASAPORT.get(...).x()`) become plain module function
calls against D1.

### Idempotency via D1 constraints, not application-layer event tables

Vote toggles, single-row upserts, etc., rely on unique constraints +
`INSERT ... ON CONFLICT DO UPDATE`. Concurrent writes to the same row are
serialised by D1 itself. The application layer does not maintain its own
event-id ledger.

### Live updates cut for now

`useLiveAgent.ts`, its consumers, and the `22-live-updates.spec.ts` e2e
test are removed. Phoenix's UX is request-response only until a product
need for real-time emerges. At that point a single, scoped DO for
WebSocket fan-out can be re-introduced under its own ADR.

### File-layout consequence

`apps/web/worker/view/` is removed. Its drizzle schemas migrate up to
`apps/web/worker/db/` (or wherever resolvers reach for them — exact path
decided during the rip; the principle is that D1 is no longer a "view").
Feature directories (`features/sozluk/`, `features/pano/`,
`features/pasaport/`) keep their `*Reader` modules and gain matching
writer modules alongside them. Mutation logic lives next to its read
shape, not behind a DO RPC seam.

## Consequences

### What gets deleted

- `worker/features/sozluk/SozlukTerm.ts` + `term-drizzle/`
- `worker/features/pano/PanoPost.ts` + `post-drizzle/`
- `worker/features/pasaport/Pasaport.ts` + `drizzle/` (durable-sqlite
  migrations and schema)
- `worker/view/PhoenixProjection.ts` and its drizzle directory becomes the
  authoritative schema
- All `runInDurableObject`, outbox assertions, and `idFromName`
  call sites in `tests/integration/`
- `src/lib/useLiveAgent.ts` and `tests/e2e/22-live-updates.spec.ts`
- DO + workflow bindings in `wrangler.jsonc`

### What survives unchanged

- React + Relay frontend, GraphQL schema and resolvers (mostly — error
  surfaces consolidate per audit candidate 3 as a separate cleanup)
- Effect `resolver()` wrapper and the per-request runtime
- The seed importer (`apps/web/scripts/seed-sozluk.ts`); it switches from
  hitting DO admin endpoints to writing D1 directly
- `*SummaryReader` modules (they were already D1 readers; they keep being
  the right shape)
- E2E tests except live-updates

### Audit candidate alignment

Of the seven candidates from `audits/2026-05-15-cross-layer-friction-and-0008-cleanup.md`:

| Candidate | Disposition |
|---|---|
| 1. Outbox-pattern remnants | Vaporises with the rip. |
| 2. Vote semantics duplicated per voteable type | Still real. Vote is now a D1 module shared across sozluk + pano. Address during or right after the rip. |
| 3. Mutation error mapping triplicated | Still real. Codec lives in the Effect `resolver()` wrapper. Address during or right after the rip. |
| 4. Test surface drifts past the interface | Mostly vaporises (no DO internals to peek at). Reader-direct tests remain a small smell to address later. |
| 5. "Coordination atom" unnamed + reader vocab inconsistent | Coordination atom concept retires with the DOs. Reader vocab is still worth pinning in `terms.md` later. |
| 6. Field rename = ~15 file edits | Mildly reduced (one fewer layer). Still a long-term watch. |
| 7. Effect services as single-adapter seams | Unchanged. Stays a watch item. |

### Pre-prod hygiene

D1 is wiped and re-seeded as part of the rip. There is no live data and
no migration script.

### Banned

- **Adding a Durable Object** without a follow-up ADR that explicitly
  re-introduces the pattern, scoped to a named use case (real-time
  fan-out, hot-row contention, cross-aggregate atomicity).
- **Adding Cloudflare Workflows** for mutations or projection. Resolvers
  do their own writes. If a long-running async operation appears, that
  operation gets its own ADR.
- **Outbox tables, projection workflows, `processed_event` ledgers** in
  the application layer. D1 constraints are the idempotency surface.
- **DO-internal sqlite, `runInDurableObject` in tests, `idFromName`
  call sites.** All are dead vocabulary.

### When to revisit

- **Real-time UX becomes a product requirement.** Re-introduce a single,
  scoped DO for WebSocket fan-out. Re-adopt the relevant parts of 0006.
  *(Triggered: [0023](0023-live-views-sse-livedo.md) introduces `LiveDO` for
  live-view fan-out. D1-direct otherwise stands.)*
- **Per-entity hot-write contention shows up on a row** (a single post
  receives concurrent vote storms that contend at D1's row lock).
  Re-introduce per-atom sharding for that entity specifically.
  Re-adopt 0005 scoped to one entity.
- **Cross-aggregate atomic mutations** become unavoidable (e.g. Künye v2
  invite-spend that has to mutate sponsor + invitee + audit log
  atomically). Re-introduce coordinator workflows for just those
  operations. Re-adopt 0008 scoped to the cross-aggregate set.
- **D1 storage or write-throughput limits** become painful at observed
  traffic. Mitigation order: tune indexes, peel a hot table into its own
  D1, then consider DOs.

The pattern: every retired ADR can be re-adopted, scoped to the specific
pressure that demands it. The default is the simplest path that ships the
product.

### Out of scope

- **Künye / achievement system architecture.** The 2026-05-10 grill log
  decided the product surface (Lobsters-shaped rep, v1 zero badges,
  agent registration). It assumed the DO+workflow stack. The product
  decisions survive; the implementation reshapes around D1 when the
  feature is picked up.
- **Replacing better-auth.** It already supports a D1 adapter; the swap
  is mechanical.
- **Multi-region or read-replica D1 strategy.** Not needed at zero
  scale.
