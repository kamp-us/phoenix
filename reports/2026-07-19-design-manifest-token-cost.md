# Per-diff token cost: whole `design-system-manifest.md` read vs scoped per-component fetch

*Dated findings — 2026-07-19. Investigation for [#3640](https://github.com/kamp-us/phoenix/issues/3640); the decision input that gates the founder-deferred [#3158](https://github.com/kamp-us/phoenix/issues/3158) (on-demand per-component contract delivery).*

**The question.** Today a `write-code` agent reads the whole `design-system-manifest.md`
before authoring any UI (Step 4a). #3158 proposes an on-demand mechanism (an MCP tool or a
structured retrieval seam) that would instead fetch only the touched component's contract
from the descriptive inventory (`design-system-inventory.md`). Is the per-diff token saving
big enough to justify building and maintaining that mechanism?

**The answer, up front: recommend-defer.** The measurement does not just come up short — it
invalidates the premise. The manifest and the inventory are **not substitutes**: the manifest
is cross-cutting design *law* every UI diff needs regardless of which component it touches, and
the inventory is a separate *descriptive* catalog that `write-code` does not read per diff at
all today. A scoped per-component fetch therefore saves **nothing** off the real per-diff cost.
The reasoning and the numbers behind that call are below.

## Method + tokenizer

- **Token counts** are the `cl100k_base` encoding (the public GPT-4/3.5 BPE, via `tiktoken`)
  run over the exact files on `main` at the time of writing. `cl100k_base` is a *reference*
  tokenizer, not Claude's — Claude's tokenizer is not public — so treat these as a defensible
  proxy; a Claude-native count for English + Markdown typically lands within roughly ±10–20%.
  Every count is also reported as a **chars/4 floor** (the widely used lower-bound heuristic)
  so the numbers stand even if you distrust the specific tokenizer.
- **Artifacts measured** (repo-relative, on `main`):
  - `design-system-manifest.md` — the whole-manifest read.
  - `design-system-inventory.md` — the scoped-fetch source (generated by
    `pipeline-cli design-inventory generate`, #3155).
- **Touched-component distribution** is derived from real history: `git log` over
  `apps/web/src/components/ui/`, counting distinct primitives touched per commit (see
  [Sampled diffs](#the-sampled-diffs-touched-component-distribution)).

## 1. Whole-manifest read cost

| Artifact | Lines | Chars | chars/4 floor | `cl100k_base` |
|---|---:|---:|---:|---:|
| `design-system-manifest.md` (whole) | 379 | 26,145 | 6,536 | **7,096** |

A `write-code` agent pays this **~7,100 tokens on every UI diff**, independent of which
primitive the diff touches, per Step 4a of the `write-code` skill (it reads the manifest whole
for the role-token annotations, component-selection rules, and per-pillar prohibitions before
authoring UI).

## 2. Scoped-fetch cost (one component's contract from the inventory)

| Artifact | Lines | Chars | chars/4 floor | `cl100k_base` |
|---|---:|---:|---:|---:|
| `design-system-inventory.md` (whole) | 331 | 15,271 | 3,817 | 3,688 |
| Inventory preamble/schema (shared, read once) | 11 | 622 | 155 | 144 |
| **One component slice** (per-primitive `##` section) | — | — | — | **67–172 (mean 107, median 100)** |

The inventory documents **33 primitives** (the `apps/web/src/components/ui/` library is ~29
`.tsx` files; several files export multiple primitives — `atoms.tsx` → `Code`/`Kbd`/`Mark`/…,
`Form.tsx` → `Field`/`FieldError`/`Label`). Per-slice token counts, `cl100k_base`:

- `Input` **67** · `Tooltip` **95** · `Card` **141** · `Button` **154** · `Menu` **167** ·
  `Dialog` **172** — mean **107**, median **100** across all 33.

A realistic scoped fetch for a diff touching *k* components reads the ~144-token
preamble/schema once plus *k* slices:

| Components touched (*k*) | Scoped cost (`cl100k_base`) |
|---:|---:|
| 1 | 144 + 107 ≈ **251** (≈ 107 if the preamble is skipped) |
| 2 | 144 + 214 ≈ **358** |
| 5 | 144 + 535 ≈ **679** |

## 3. The sampled diffs (touched-component distribution)

`git log` over `apps/web/src/components/ui/` (last ~40 commits touching the directory),
counting distinct primitives changed per commit. The raw counts are dominated by a handful of
**mechanical/bulk** commits that are *not* representative of a hand-authored UI feature/fix
diff — a JSDoc-convention rollout (14 and 9 primitives), a repo-wide biome format (12), a
comment-deslop pass (4), and the original design-system import (11). Excluding those, the
**19 representative hand-authored UI feature/fix diffs** distribute as (estimate, from the
sampled window):

- **median = 1** touched primitive; **mean ≈ 1.6**; **p90 ≈ 4–5**.
- Fourteen of nineteen touched exactly one primitive; the multi-component outliers were real
  refactors (composite-primitive extraction: 4; type-ramp recalibration: 3; systematic focus
  layer: 5; a11y/motion globals: 4).

So the **typical** UI diff touches one primitive; the realistic worst case for a single
feature/refactor diff is ~5. *(Labeled an estimate: it is one sampled window of history, and
the mechanical/representative split is a judgment call — but the median-of-1 signal is strong
and unlikely to move.)*

## 4. The delta per diff — and why it is the wrong number

Taken at face value, the delta looks enormous: **~7,100 (whole manifest) − ~250 (scoped, k=1)
≈ ~6,850 tokens/diff saved**, a ~96% reduction; even at k=5 the scoped cost (~680) is a ~90%
cut. If the two reads were substitutes, #3158 would be an easy build.

**They are not substitutes — this is the finding that decides #3158.** Reading the actual
files on `main` shows the whole-manifest-vs-scoped-slice framing rests on a category error:

1. **The manifest is invariant design *law*, not a per-component catalog.** Its sections are
   the four pillars + their prohibitions, the founder-ratified v1 design values, the semantic
   role-token annotations, the component-*selection* rules, the navigation-IA taxonomy + the
   machine-checkable placement law. Only one 7-row table ("component-selection rules") is even
   component-shaped, and it is about *which composite shell to reach for*, not a per-primitive
   contract. **Every UI diff needs this law regardless of which primitive it touches** — you
   cannot replace a ~7,100-token cross-cutting-law read with a ~107-token slice of one
   component without dropping the pillars, role tokens, and prohibitions the diff must satisfy.
2. **The inventory is a separate *descriptive* catalog that `write-code` does not read per diff
   today.** The `write-code` skill references `design-system-manifest.md` (Step 4a) and never
   references `design-system-inventory.md`; the inventory's own preamble states it is
   "DESCRIPTIVE ONLY … The normative design law … is founder-authored in
   `design-system-manifest.md` and is never written here." So there is **no whole-inventory
   read cost in the pipeline today to scope down.** #3158 would capture a saving against a cost
   the pipeline does not currently pay.

Put together: **the scoped per-component fetch delivers ~0 saving against the real per-diff
cost (the manifest), because the manifest is not per-component-sliceable and the inventory is
not read per diff.** The only world where #3158 saves tokens is a *hypothetical future* one
where `write-code` reads the **whole inventory** on every diff — and even there the whole
inventory is only **~3,700 tokens** (roughly half the manifest, well under 2% of a coder's
context budget), so scoping it would save ~3,400 tokens/diff at k=1: real, but small in
absolute terms and *additive to*, not *reducing*, the dominant manifest cost.

## 5. How it compounds as the manifest + `components/ui` grow

The growth model matters because it is the one argument for building ahead of need — and it
does not hold up:

- **Whole-manifest read cost is O(design law), not O(component count).** The manifest grows
  when the founder adds cross-cutting law (a new pillar clause, a new token role, a new IA
  rule) — *not* linearly with each new primitive, because primitives do not get their own
  manifest section. Adding the 30th, 40th, 50th primitive leaves the manifest essentially
  unchanged. A per-component retrieval mechanism does **nothing** to bound manifest-growth
  cost, because there is no per-component manifest content to scope away. The lever for
  manifest cost is *keeping the founder-authored law terse*, not building a fetch seam.
- **Scoped-fetch cost is O(*k*), flat in library size.** It scales with how many components a
  diff touches (~1 typically), never with how many exist — a genuinely nice property, but one
  that only pays off against a whole-*inventory* read that the pipeline doesn't do.
- **The inventory grows O(component count)** (~107 tokens/primitive). At today's 33 primitives
  it is ~3,700 tokens; a 3× library (~100 primitives) would put the whole inventory near
  ~11k tokens. Only if `write-code` were *also* changed to read the whole inventory per diff
  would scoping become worth the mechanism — and that is two contingent changes away, neither
  of which is on the roadmap.

## Recommendation

**Recommend-defer building #3158's delivery mechanism (an MCP tool / structured retrieval
seam).** The measurement invalidates its justifying premise: the whole-manifest read (~7,100
tokens/diff) is invariant design *law* that a scoped per-component fetch cannot replace, and
the inventory it would fetch from (~3,700 tokens whole) is not read per diff today — so the
scoped mechanism saves ~0 tokens against the real per-diff cost. A new stateful delivery seam
carries a real fixed cost (the tool itself, a freshness contract, a CI guard to keep it in
sync — the shape #3156's guard already had to take for the inventory), and a saving that is
zero-against-actual-cost and, even in the most generous hypothetical, ~3.4k tokens/diff against
a cost the pipeline doesn't pay, does not justify it.

**Cheaper moves that dominate the mechanism, if per-component context is ever wanted:**

1. Keep `design-system-manifest.md` **terse** — this is the only lever that actually bounds the
   dominant per-diff cost, and it needs no mechanism.
2. If an agent ever needs one component's descriptive contract, point it at the single relevant
   `##` section of `design-system-inventory.md` via a plain path + anchor (a ranged `Read`) —
   scoped retrieval with **zero new machinery**, no MCP server, no retrieval seam.

**Revisit trigger (when to re-measure).** Re-run this measurement only if **both** hold: (a)
`write-code` is changed to read the **whole** `design-system-inventory.md` per diff, **and** (b)
the inventory grows past ~15–20k tokens (roughly a 4–5× library expansion). Until both are true,
#3158's mechanism is unjustified and should stay deferred.
