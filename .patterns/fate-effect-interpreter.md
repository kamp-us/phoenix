# fate-effect interpreter — `FateInterpreter` + the protocol codecs + the differential oracle

The v2 native plane of `@phoenix/fate-effect` (PRD story 16; ADR 0043): fate's `handleRequest` reimplemented as an Effect program, developed **against the v1 compiled server as a differential oracle** — identical operations through both backends must produce byte-equal wire output. **This is the SERVING path**: since the cutover (ADR 0043) the worker's `POST /fate` route yields `FateInterpreter.handleRequest` on the request fiber ([fate-effect-worker-wiring.md](./fate-effect-worker-wiring.md)). The full operation surface is oracle-green — query/mutation/custom-list dispatch, the byId plane (the Effect selection walk over `RequestResolver`-batched sources, `Walk.ts`), and the connection plane (Schema-decoded pagination args + fate's in-array windowing, `Connection.ts`) — across sozluk-, pano-, pasaport-, and stats-shaped fixtures; the compiled v1 server remains as the oracle's baseline only.

## Serving (`route.ts` — the platform seam)

- **No runtime on the request path**: the route yields the program; the platform layer (alchemy's worker bridge running the compiled `HttpRouter.toHttpEffect`) owns the one run boundary. The package-side `runtime.runPromise` in `Executor.ts` is oracle-baseline-only (the conversion-point enumeration in `Executor.test.ts` still pins it to exactly one site).
- **Spans**: handler/source `Effect.fn` spans parent to the router's request span (the `HttpEffect.toHandled` tracer middleware) because everything runs in the request fiber's tree — including loads through the walk's `RequestResolver` batch fiber. Pinned in `Interpreter.batch.test.ts` § observability with a `ParentSpan` collector.
- **Abort**: the interpreter deliberately leaves interruption to the caller. The route wires `request.signal` → fiber interruption via `interruptOnAbort` (effect-smol's `HttpEffect.toWebHandlerWith` idiom), because alchemy's bridge passes no signal. The served `FateRequestContext` carries no `signal` field at all (audit fix A1); the only signal-bearing context is the oracle baseline's Executor-local `ExecutorRequestContext` (`Executor.ts`).

## The protocol codecs (`Protocol.ts`)

Two codec surfaces, split the way fate itself splits them:

- **The canonical schemas** — `ProtocolOperation`, `ProtocolRequest`, `ProtocolError`, `ProtocolSuccessResult`/`ProtocolFailureResult`/`ProtocolOperationResult`, `ProtocolResponse` — are fate's exported protocol types in Effect Schema form, field for field. Response structs declare fields in **fate's serialization order** (`{data, id, ok}` / `{error, id, ok}` / `{code, issues, message}`); Schema encode emits keys in declaration order, which is what makes byte-equality achievable.
- **`decodeProtocolRequest`** is fate's `assertProtocolRequest` as a **staged** Schema decode (envelope → per-operation base → kind-conditional fields), each stage failing with fate's own `FateRequestError("BAD_REQUEST", <fate's exact message>)`. The staging preserves fate's **leniency** as much as its strictness: fate checks `ids`/`type` only for `byId` and `name` only for named kinds, so junk in kind-unchecked fields must pass — the canonical schema is *stricter* than fate's runtime gate and is therefore not the dispatch gate. The decoded result is a discriminated union (`ProtocolByIdOperation | ProtocolNamedOperation`), so dispatch cannot see an unvalidated state.

**The drift pin** (`Protocol.unit.test.ts`): type-level `toEqualTypeOf` pins of the schemas' `Type` against `@nkzw/fate`'s exported `FateOperation`/`FateProtocolRequest`/`FateProtocolResponse` (+ result/error members derived from them), under a `Wire<>` normalizer that erases **mutability only** (fate types mutable arrays; Schema types `ReadonlyArray` — indistinguishable on the wire). Keys, optionality, and value types stay exact, so a fate upgrade that moves the protocol fails `tsgo` before any runtime test runs. The wire `code` is deliberately `string` (phoenix's annotated codes are wider than fate's closed union — [fate-effect-wire-errors.md](./fate-effect-wire-errors.md)); the pin substitutes fate's union back in and demands exact equality on everything else.

## The dispatch loop (`Interpreter.ts`)

`FateInterpreter.handleRequest(request, context): Effect<Response, never, FateServer>` — decode → run concurrently → encode:

- Operations dispatch via `Effect.forEach(…, {concurrency: "unbounded"})` (the order-preserving collector; fate's own loop is `Promise.all`).
- Each operation runs the entry's `resolve` through the **one shared provision pipeline** — `provideRequestPair(context, server.services)` in `Provision.ts`, built once per request and applied per operation: the per-request pair provided as VALUES off the `FateRequestContext` (innermost, so request values win), the captured build-time services beneath. The same helper is the walk's per-request `provide` and the v1 compiler's `runResolve` pipeline (oracle byte-equality proved the unification), and it carries the package's ONE `erased→kernel` `R: unknown → never` re-pin — no call site spells the cast anymore. Minus the Promise hop: **no runtime is owned here**. The caller runs the program (the route's request fiber in production; the oracle's test `ManagedRuntime` in the suite), so the package's single Effect→Promise conversion point stays in `Executor.ts`, oracle-baseline-only. The helper's own pins (`Provision.unit.test.ts`) cover provision order, request-values-win (expressible there through Context contravariance — vacuous at every call site, since `FateServerRequirements` excludes the pair), and the re-pin's types.
- Failures map through `encodeWireError` — the **one annotation codec both backends share** — then onto the protocol error shape; request-level failures (malformed JSON/protocol) serialize as fate's single `id: "request"` result with the error's own status. (The raw legacy config arms were removed with the cutover — every record is constructor-built, so there is no un-interpreted shape left.)
- Lookups guard with `Object.hasOwn` — fate indexes the raw record and would trip over prototype names (`"constructor"`); a prototype name is not a registered operation. Deliberate, documented divergence; everything else is byte-faithful.

## The selection walk + batched sources (`Walk.ts`)

fate's byId plane (`executeOperation`'s byId arm → `resolveSourceByIds` → `resolveNode`/`filterToViewFields`/`toConnectionResult`) as an Effect program. `makeWalk(server, context)` is constructed **once per `handleRequest`, before the dispatch loop** — its `RequestResolver` instance IS the batch window, so concurrent operations in one protocol request share it and the window never spans requests.

**Mirrored byte for byte** (the walk oracle enforces it): the dispatch order (falsy `type` → fate's `BAD_REQUEST`; source lookup → `NOT_FOUND`; `ids.map(String)` so numeric wire ids coerce before the source); the selection plan (`"id"` always selected, resolver/computed fields at terminal segments only, a selected ref auto-selects the child view's `id`, unknown paths ignored); `resolveNode` (authorize falsy → `null` with resolve skipped, `undefined` results stay absent, computed deps derive from the `select` declaration, relations recurse into records/arrays/shaped connections); and `filterToViewFields` — masking emits keys in the **view's field declaration order**, the byId plane's serialization-order mechanism (the named planes get theirs from the `Protocol.ts` structs).

**The error taxonomy splits in two**, matching fate exactly: a view callback (resolver/computed `authorize`/`resolve`) throwing a `FateRequestError` passes through verbatim, and any other callback throw is fate's *own* `toProtocolError` arm (`INTERNAL_ERROR` / "Internal server error.") — these never pass through `encodeWireError` in fate, so the walk masks them itself. Source-handler defects, by contrast, ride the operation's exit as raw causes and collapse through `encodeWireError` exactly like the v1 compiled executors (`INTERNAL_SERVER_ERROR` / "Something went wrong."). A capability-less source (the `contributionSource` shape) is fate's internal arm with the fixed message.

**The batching contract** (PRD story 17 — where N+1 dies): one `Request.Class` value per byId operation (`{source, ids}`); `runAll` groups entries by source-entry identity, unions ids (first-arrival order, **deduplicated** — the dedupe AC), and mirrors fate's two executor arms. `byIds`-capable: ONE call with the union; a lone operation gets the rows verbatim, a merged window masks per operation by primary key **in source-return order** — byte-equal to fate whenever the source is *membership-stable* (rows are a function of the id SET; every SQL `IN`-shaped loader, i.e. every phoenix source, qualifies — this is the loader contract's batching corollary). `byId`-only: one call per unique id; each operation reassembles its own ids (order and duplicates kept, nulls dropped — fate's `flatMap` arm); a failed id fails exactly the operations that asked for it. Loads self-provide the per-request pair + captured services (the shared `provideRequestPair`, `Provision.ts`), so the resolver fiber needs no ambient context.

**Deliberate divergences** (documented in `Walk.ts`, pinned in the suite): the interpreter resolves sources from `config.sources` directly, whereas fate populates `sourcesByType` only by visiting ROOT views — the v1 compiled server passes `roots: {}` (ADR 0016/0019), so its byId plane is unreachable dead code. The divergence is strictly additive on the wire and **LIVE since the cutover** (ADR 0043's recorded user-visible change): fate's client CAN emit `kind: "byId"` (cache-miss node fetches, missing-field refetches, and the live-payload fallback in `fetchLiveRecord`), and v1 served all of those `NOT_FOUND` — byId went error→data, which **fixes a latent live-refetch breakage** rather than introducing one. fate's hidden computed-state stamping is unreachable for package-authored loaders and is skipped.

## The connection plane (`Connection.ts`)

fate's `paginationArgsSchema`, `arrayToConnection`, and `getScopedArgs` on Effect Schema — this removes **fate's only runtime zod** from phoenix's execution path (the PRD's "removes fate's only runtime zod usage from our graph").

- **The accept/reject boundary is fate's, exactly.** Only the four pagination keys (`after`/`before`/`first`/`last`) are extracted from the scoped args bag before the schema sees it (fate's `extractPaginationArgs` → `z.strictObject` flow — feature args never trip strictness). `after`/`before` must be strings; `first`/`last` positive integers; the two refines reject `after`+`before` and `first`+`last` with fate's messages and fate's TRUTHY predicates (`!(after && before)` — an empty-string cursor passes). A rejected bag is fate's masked internal arm on the wire (`INTERNAL_ERROR` / "Internal server error." — in fate the zod throw rides `executeOperation`'s catch into `toProtocolError`), NOT the annotation codec's arm.
- **Windowing defaults are fate's, exactly**: no pagination keys → every node, no windowing, all-false pagination; otherwise page size `first ?? last ?? nodes.length`, cursors found IN the array (`String(node.id)`), unknown cursor → full array (`findIndex < 0`).
- **Scoping**: `Walk.ts`'s `toConnectionResult` threads the OPERATION args and the running dotted path through every recursion; a raw array under a selected list-kind field wraps via `arrayToConnection` with `getScopedArgs(args, path)` (`args: {chapters: {first: 2}}` windows `chapters`; top-level keys never leak into nested windows). Already-shaped connection envelopes (resolver-delivered, the ADR 0019 idiom) pass through, recursing per entry node — the walk **never auto-fetches** a resolver-owned connection (a row without the field masks to nothing, matching production semantics in [fate-connections.md](./fate-connections.md)).
- **What is deliberately NOT reimplemented**: fate's `resolveConnection`/`resolveSourceConnection` (the root-list keyset plane, default page size 20) is reachable only through `createFateServer`'s `rootLists`, and phoenix configures no roots (ADR 0016/0019 — lists are resolver-handled, keyset cursors owned by domain services). The interpreter's list plane is custom lists; unknown list names are `NOT_FOUND` on both backends. A `Fate.source` `connection` handler therefore stays exactly as reachable under v2 as under v1: not at all.

## The differential oracle (`Oracle.fixture.ts` + the `Interpreter*.test.ts` planes)

The dual-stack harness and the ONE sozluk fixture world live in `Oracle.fixture.ts` (shared with `Executor.test.ts`, the baseline's own pin suite); the corpus is split one test file per plane: `Interpreter.test.ts` (harness smoke + the sozluk operation corpus, malformed-protocol rejections included), `Interpreter.walk.test.ts` (byId + nested refs + the connection plane, walk world co-located), `Interpreter.features.test.ts` (the pano/pasaport/stats shapes, fx world co-located), and `Interpreter.batch.test.ts` (RequestResolver batching, concurrent dispatch, observability, and the pinned byId divergence). The harness runs every corpus step through **both** backends — v1 (`FateExecutor.toFetchHandler` over a `ManagedRuntime`) and v2 (`FateInterpreter.handleRequest` through its own runtime) — each over its **own fresh in-memory database**, and asserts equality of status, content-type, body **text** (bytes, not parsed shape), and the recorded `LivePublisher` calls. Mutations run through both worlds in lockstep, so later reads prove **state parity**, not just response parity.

Corpus rules learned here:

- **No intra-batch read-after-write.** fate's `Promise.all` and Effect's fiber scheduling interleave differently; neither backend guarantees write-then-read ordering inside one protocol request. Only cross-request reads are deterministic — the oracle only pins deterministic output.
- Error cases are corpus citizens: annotated codes, `UNAUTHORIZED`, `VALIDATION_ERROR` (same Schema renders the same message on both sides), `NOT_FOUND`, defects (fixed internal message), died-`FateRequestError` passthrough with `issues`, dispatch-time `BAD_REQUEST` (empty name), and every malformed-protocol rejection with its 400 status.
- fate's acceptance **leniency** is itself a corpus case (junk in kind-unchecked fields must succeed on both sides).
- **The walk corpus needs its own v1 baseline.** The package's v1 server cannot serve byId at all (`roots: {}` → empty `sourcesByType`), so the walk steps run against fate's *own* `createFateServer` over the SAME compiled source executors (`compileFateSources`), with `list()`-wrapped roots whose only job is registering sources by type (a `list()` root demands no matching resolver — fate's loop). That baseline is fate's real dispatch + walk, which is the thing byte-equality must pin. The v1↔v2 byId divergence itself (v1 `NOT_FOUND`, v2 serves from `config.sources`) is pinned as an explicit non-parity test so it can never become an accident.
- The batching ACs are **v2-only counting tests** (a wrapping source records its calls): N same-entity byId operations in one request → exactly one `byIds` call; duplicate ids deduped before the source; byId-only sources load each unique id once; two sequential requests → two calls (the window is the request, not the runtime).
- **Cursor round-trips are read off the wire, not hardcoded**: page 1's `nextCursor` is parsed from the BASELINE's response and fed as page 2's `after`, with both pages byte-compared — for the walk's nested connections, for resolver-owned inline connections, and for custom keyset lists (the ADR 0019 lockstep at oracle level).
- **The feature-shaped corpus** pins each migrated feature's distinctive operation shapes in-memory (not D1): pano (embedded-scalar `tags` through both the resolver and the walk's mask, threaded comments as a resolver-owned inline keyset connection, a keyset feed list, a publishing mutation), pasaport (`profile` stamping `id` === `userId`, the resolver-owned `contributions` connection, the capability-less `Contribution` source → fate's internal arm), and stats (plain string-typed queries with no data view). Feature byId steps reuse the walk-baseline arrangement over the same config.

The full operation surface is byte-equal; the route serves the interpreter (ADR 0043), and the oracle remains the regression net — extend the corpus when the surface grows.
