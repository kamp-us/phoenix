# The stack & deploy

> Derived from `alchemy@2.0.0-beta.59` ‚Äî re-verify on pin bump.

How phoenix is declared and shipped. The short answer: `alchemy.run.ts` is an Effect program ‚Äî an `Alchemy.Stack` ‚Äî that declares the resources (the D1 database, the worker) and returns the stack's outputs. `alchemy deploy` runs it against the Cloudflare API; `alchemy dev` runs it locally. There is no `wrangler.jsonc`.

## The stack

```ts
// apps/web/alchemy.run.ts
import * as Alchemy from "alchemy";
import * as Cloudflare from "alchemy/Cloudflare";
import * as Effect from "effect/Effect";
import {resolveStateMode} from "./worker/env.ts";
import PhoenixLive, {Phoenix} from "./worker/index.ts";

export default Alchemy.Stack(
  "phoenix",
  {
    providers: Cloudflare.providers(),
    state:
      resolveStateMode(process.env) === "cloudflare" ? Cloudflare.state() : Alchemy.localState(),
  },
  Effect.gen(function* () {
    const worker = yield* Phoenix;
    return {url: worker.url};
  }).pipe(Effect.provide(PhoenixLive)),
);
```

- **`providers`** ‚Äî the resource implementations the stack may use. `Cloudflare.providers()` covers Workers/D1/DOs/assets ‚Äî phoenix's full surface. (There is no `Drizzle.providers()` in the stack: migration SQL is generated by `drizzle-kit` out-of-band; alchemy only applies it via the D1 resource's `migrationsDir`. See [alchemy-drizzle-d1.md](./alchemy-drizzle-d1.md).)
- **`state`** ‚Äî where alchemy stores the deployed-resource state it diffs against. `Cloudflare.state()` is a **Cloudflare-hosted** store (reproducible across machines/CI, but its Layer needs credentials + network on *every* run, including `alchemy dev`); `Alchemy.localState()` is a **file-based** store under `<cwd>/.alchemy/state/<stack>/<stage>/` ‚Äî gitignored, so per-checkout *and per-worktree* (`alchemy@2.0.0-beta.59 ‚Äî src/State/LocalState.ts`). phoenix picks the store from a dev-vs-deploy signal (`resolveStateMode` in `worker/env.ts`), **not** from `CI` ‚Äî `CI` is set for both the deploy workflow and the integration-test job and can't distinguish them. Only `alchemy dev` resolves to `localState()`; a real `alchemy deploy` ‚Äî and the `Test.make` integration harness, which deploys to real remote Cloudflare (ADR [0082](../.decisions/0082-two-test-tiers-unit-integration.md)) ‚Äî uses the shared Cloudflare store. **"Local state" does not mean "local resources":** even under `alchemy dev`, the D1/KV/DO resources the stack declares are created on real Cloudflare in your personal stage ‚Äî only the worker runs locally (ADR [0032](../.decisions/0032-alchemy-beta45-and-dev-model.md); see [Stage lifecycle & state discipline](#stage-lifecycle--state-discipline) for why that makes the local state file precious).
- **body** ‚Äî yields the resources to deploy and returns the stack outputs (the deployed URL, etc.).

Yielding `Phoenix` deploys the worker; the worker's own init phase (its `bind()` calls and `DurableObjectNamespace` declarations) tells alchemy which bindings, DOs, and migrations to send. The stack doesn't re-declare them.

## Resources shared between stack and worker

Resource *declarations* the worker binds ‚Äî the D1 database ‚Äî live in a module both import, so there's one definition:

```ts
// apps/web/worker/db/resources.ts
export const PhoenixDb = Cloudflare.D1.Database("phoenix_db", {
  migrationsDir: "./worker/db/drizzle/migrations",
  migrationsTable: "drizzle_migrations",
});
```

Migration SQL under `migrationsDir` is generated out-of-band by `drizzle-kit` (against `worker/db/drizzle.config.ts`) and committed; there is no `Drizzle.Schema` resource. `migrationsTable` matches drizzle-kit's bookkeeping table so the applied-set is shared between generator and applier. See [alchemy-drizzle-d1.md](./alchemy-drizzle-d1.md).

> **`migrationsDir` ‚âÝ adopting a DB.** `migrationsDir` only tells alchemy where the migration SQL lives, not which physical DB binds; `migrationsDir` migrations are idempotent (already-applied skipped, new ones applied ‚Äî `migrationsTable: "drizzle_migrations"` for Drizzle compatibility). Which physical DB binds is decided by state + the adoption probe: `Cloudflare.D1.Database` *can* re-link an existing DB by name lookup on a state-less deploy, **but only when the resource carries a stable explicit `name` prop** ‚Äî an auto-named D1 gets a random per-instance suffix the lookup can never recompute. phoenix pins such a `name` for `PhoenixDb` **on the local-state dev path only** so the dev loop re-adopts (auto-named on hosted state, where naming a tracked DB would be a destructive `replace`; #2361). The full semantics live in [Stage lifecycle & state discipline](#stage-lifecycle--state-discipline).

The worker `bind()`s `PhoenixDb` ([alchemy-bindings.md](./alchemy-bindings.md)); the stack ensures the DB exists and its committed migrations have been applied before the worker deploys. The Durable Objects (`ConnectionDO`, `TopicDO`) are the `DurableObjectNamespace` classes themselves ‚Äî declaring them and binding them in the worker is enough; alchemy derives the migrations. Under the hood alchemy tracks DO migrations via Cloudflare **worker tags** (`alchemy:do:{id}:{class}`, `alchemy:migration-tag:{version}`), **defaults new DO classes to `new_sqlite_classes`**, and computes `deletedClasses` by diffing the old-vs-new tag maps.

> **Cutover went fresh on both axes (no data to preserve).** phoenix is effectively greenfield: prod ran but the D1 held no irreplaceable data ([ADR 0009](../.decisions/0009-d1-direct-defer-dos-and-workflows.md)). So:
> - **Fresh D1** ‚Äî `Cloudflare.D1.Database` creates a new database and applies migrations. (The old `import-sozluk`/`import-pano` seeders were deleted as a fail-open security hole; there is no seeding mechanism and none is planned ‚Äî see CLAUDE.md's S√∂zl√ºk-seed guard.)
> - **Fresh worker script name** ‚Äî deploying under a new name avoids two hazards of reusing the wrangler-managed `phoenix` script: (1) alchemy brands a script with no `alchemy:*` tags as **`Unowned`** and refuses to deploy without an explicit `--adopt`; (2) the DO migration-tag precondition won't match wrangler's existing tags (alchemy *does* have a self-healing recovery that parses the expected tag from the error and retries, but a fresh name skips the dance entirely).
> The old wrangler-managed `phoenix` worker + `phoenix_db` were deleted once the alchemy worker was verified. Reusing the same script name is technically supported (`--adopt` + the tag-recovery path) but strictly higher-risk for zero benefit, since there's no data to keep.

## What `wrangler.jsonc` keys become

| `wrangler.jsonc` | alchemy |
|---|---|
| `name`, `main`, `compatibility_*` | `Cloudflare.Worker` id + props ([alchemy-worker.md](./alchemy-worker.md)) |
| `assets` | the `assets` prop on the worker |
| `d1_databases` | `Cloudflare.D1.Database(...)` resource |
| `durable_objects.bindings` | `Cloudflare.DurableObjectNamespace` declarations |
| `migrations` (DO classes) | derived from the DO declarations |
| `d1 migrations_dir` | `D1Database({migrationsDir, migrationsTable})` ([alchemy-drizzle-d1.md](./alchemy-drizzle-d1.md)) ‚Äî SQL generated by `drizzle-kit` out-of-band |
| `vars` | resource props / `Layer.succeed` config |

## Dev & deploy

```bash
pnpm build        # vite build ‚Üí dist/client (the SPA assets the worker serves)
alchemy deploy    # bundle the worker, upload dist/client, push to the Cloudflare API
alchemy deploy --stage prod
```

The SPA is built by Vite as a normal build step (`dist/client`), then uploaded via the worker's `assets` prop ‚Äî `alchemy deploy` does not drive Vite for phoenix's single-worker shape ([alchemy-worker.md](./alchemy-worker.md) explains why it's `Cloudflare.Worker` + `assets`, not `Cloudflare.Vite`). Drop `@cloudflare/vite-plugin` from `vite.config.ts` (alchemy is incompatible with it); keep `react()` and the `fate()` codegen plugin, which reads the server's `Entity<>` types regardless of deploy path (see [fate-effect-worker-wiring.md](./fate-effect-worker-wiring.md)).

### Local dev ‚Äî two processes (the decided model)

phoenix's dev loop is **`vite dev` for the SPA (HMR) + `alchemy dev` for the worker, with Vite proxying the API**. This is a deliberate choice, not a workaround ‚Äî here's the constraint that forces it:

alchemy has **two non-mixable worker runtime paths**. The Effect-native path (`Cloudflare.Worker` + `bind()` + the Effect DO model ‚Äî what phoenix uses) runs under `alchemy dev`'s own local workerd runtime: it watch-rebuilds the *backend* on change but serves `dist/client` **statically, with no client HMR**. Integrated HMR exists only on the *other* path ‚Äî `Cloudflare.Vite` driving `@distilled.cloud/cloudflare-vite-plugin` (alchemy's own fork) ‚Äî but that path's worker entry is a plain `export default {fetch}`, so it **can't host phoenix's Effect-native worker**. You can't get `bind()`/Effect-DOs *and* integrated HMR on one worker.

> **Don't try to "fix" this by adding a Vite plugin to the Effect-native worker.** The reason there's no single-process HMR isn't a missing config ‚Äî it's the two-path split above. `@distilled.cloud/cloudflare-vite-plugin` only runs the `Cloudflare.Vite` (plain-handler) path. phoenix chose the Effect-native worker and takes the two-process dev loop as the price.

So `alchemy dev` runs the one worker (live D1/DO bindings, backend watch-rebuild), and Vite serves the SPA with HMR and proxies the API to it. `alchemy dev` serves the worker **vhost-routed** at `http://<worker-name>.localhost:1337` ‚Äî and since Node can't resolve `*.localhost`, the Vite proxy must target the IP and force the `Host` header (a `target: "http://<name>.localhost:1337"` fails with `ENOTFOUND`):

```ts
// vite.config.ts ‚Äî dev only; alchemy owns deploy
const worker = {
  target: "http://127.0.0.1:1337",        // alchemy dev's local proxy port
  changeOrigin: false,
  headers: {host: "phoenix.localhost"},   // route to the worker by its vhost name
};
server: {proxy: {"/api": worker, "/fate": worker}}; // /fate/live SSE streams through fine
```

```bash
alchemy dev     # one worker + DOs + D1, vhost at http://phoenix.localhost:1337 (local state; the D1 it binds is a REAL cloud D1 in your dev stage)
pnpm vite dev   # SPA + HMR + the fate() codegen plugin; proxies /api, /fate ‚Üí the worker
```

> **How the dev loop behaves** (alchemy `2.0.0-beta.59`, effect v4). End-to-end in a browser: `alchemy dev` runs the Effect-native `Cloudflare.Worker`; an SSE stream (`HttpServerResponse.stream`) flows through the Vite proxy live; and **editing a React component does not drop the SSE connection** (the stream runs unbroken across edits ‚Äî the proxied SSE socket isn't on Vite's HMR module graph). Three gotchas baked into the config above: (1) **`alchemy dev` is local-worker, not local-infrastructure** ‚Äî only the Worker (and DO namespaces, and Queues) has a local dev provider; D1 has none, so `alchemy dev` creates/binds a *real* cloud D1 in your personal stage and the local workerd reaches it as a remote binding (`alchemy@2.0.0-beta.59 ‚Äî src/Cloudflare/Workers/LocalWorkerProvider.ts`, the `case "d1": D1.remote(...)` arm; ADR [0032](../.decisions/0032-alchemy-beta45-and-dev-model.md)) ‚Äî `Alchemy.localState()` keeps the *state* file local and skips `alchemy login`'s hosted-store round-trip, it does not make the loop offline; (2) the `127.0.0.1` + `Host` proxy shape is mandatory (`*.localhost` is unresolvable in Node); (3) the install peers are `effect@beta` (v4; npm `latest` is v3), `@effect/platform-node`, `@effect/platform-bun`.

It stays **one worker** ‚Äî the second terminal is just the Vite dev server (not a worker); at deploy `vite build` produces `dist/client` and the single worker serves it. React HMR comes from `vite dev`, backend hot-reload from `alchemy dev`, so the full loop is live. A simpler one-terminal fallback is `alchemy dev` + `vite build --watch` (the static asset server picks up rebuilt `dist/client` on full reload, no HMR).

> **Two honest costs vs the old `@cloudflare/vite-plugin` loop.** (1) Two terminals instead of one command. (2) In dev the SPA is served by `vite dev` and routes to the worker through the Vite proxy, so dev routing is proxy rules rather than the prod `runWorkerFirst`/`assets` precedence ‚Äî a dev/prod fidelity gap to keep in mind when debugging routing. Both are accepted tradeoffs of the Effect-native single-worker choice, not blockers.

> **Session-gated SSE works through the proxy.** The better-auth session cookie survives the proxy and `/fate/live` validates it via `EventSource(..., {withCredentials:true})` ‚Äî no proxy or auth config change needed. It works because phoenix's better-auth (1.6.10) emits a dev cookie with **`Secure=false`, no `Domain`, `SameSite=Lax`, `HttpOnly`** (host-only on the Vite origin, stored over `http://localhost`, forwarded through the proxy). Two caveats this surfaced: (a) **don't flip `Secure` on in dev** ‚Äî setting `NODE_ENV=production`, an `https://` `baseURL`, or `advanced.useSecureCookies:true` makes the cookie un-storable on `http://localhost` and breaks the dev loop; (b) the worker sees `Host: 127.0.0.1:<port>` (the proxy socket), not `<name>.localhost`, so any better-auth `baseURL`/`trustedOrigins` derived from the inbound Host header would read the proxy address in dev ‚Äî set those explicitly rather than inferring from Host.

### Tests run on `alchemy/Test`, not `vitest-pool-workers`

The test harness is **`alchemy/Test/Vitest`** because **`@cloudflare/vitest-pool-workers` cannot run an alchemy worker.** The pool loads the worker module inside workerd, but the alchemy `Cloudflare.Worker` class is a deploy-time *Resource* that transitively imports alchemy's bundler (`rolldown`'s native `.node` binding + Node-only modules) ‚Äî workerd can't load it (the real workerd entry is a virtual module alchemy generates at bundle time). It's not a config problem (the pool *can* run without `wrangler.jsonc`, via inline miniflare options) ‚Äî the alchemy worker simply isn't loadable that way.

The harness deploys the stack to **real remote Cloudflare** in `beforeAll` (ADR [0082](../.decisions/0082-two-test-tiers-unit-integration.md)), then asserts over HTTP against `stack.url`. It deliberately overrides two `Test.make` defaults (`alchemy@2.0.0-beta.59 ‚Äî src/Test/Core.ts` defaults `stage: "test"` and `localState()`): phoenix passes `state: Cloudflare.state()` (durable, shared) and run-unique `it-*` stage names (`tests/integration/_stage-name.ts`), and tears the stage down in `afterAll(destroy)` unless `NO_DESTROY` keeps a stable-named stage alive for a local iteration loop (ADR [0104](../.decisions/0104-two-mode-integration-test-tier.md)). Each of those overrides is load-bearing: the default `test` stage + local state is exactly the shape that minted the orphaned `phoenix-phoenix-db-test-*` D1 cohort (#2340) ‚Äî see [Stage lifecycle & state discipline](#stage-lifecycle--state-discipline). The integration suite was ported as: `SELF.fetch(...)` ‚Üí `HttpClient` against `stack.url`; direct `env.PHOENIX_DB.prepare(...)` read-back assertions ‚Üí black-box HTTP via a worker route; and the two DO-internal tests (`runInDurableObject`/alarm, esp. `fate-live.test.ts`) ‚Üí black-box SSE/timing.

> **Tooling note.** alchemy's own examples run under `bun`; phoenix runs under `pnpm` + node. alchemy is plain TypeScript, so `pnpm alchemy deploy` / a node-run entry works. The `package.json` scripts, `turbo.json`, and CI run the alchemy toolchain; `wrangler` is gone.

> **Version pinning.** The workspace catalog pins exact versions: `alchemy 2.0.0-beta.59`, `effect 4.0.0-beta.92` (with the matching `@effect/platform-*` / `@effect/vitest` betas), and `drizzle-orm`/`drizzle-kit 1.0.0-rc.4` (`pnpm-workspace.yaml` is the source of truth). These are exact pins, not floating `alchemy@next` / `effect@beta` tags. If they ever bump, **re-verify the DO migration behavior on the new versions** ‚Äî the tag-diffing/`new_sqlite_classes` logic above was confirmed at earlier betas. The in-repo patch (`patches/alchemy@2.0.0-beta.59.patch`, ADR 0038) is re-keyed on every alchemy bump; at beta.59 it carries ONE hunk ‚Äî the Flagship `Flag` provider reconciles only the IaC-owned schema (`{variations, description}`) and never clobbers dashboard-owned serving config (ADR 0106). (The old beta.56 `DurableObjectNamespaceScope` hunk is moot: beta.59 rewrote the DO authoring model and removed the type it targeted.)

## Stage lifecycle & state discipline

A **stage** is an isolated instance of the stack ‚Äî `--stage <name>` deploys an independent copy (its own worker, its own D1, its own DOs); this is how preview deploys work without a second config file. Everything below is the lifecycle contract around that isolation ‚Äî the part that, ignored, mints orphaned cloud resources (#2340).

**Stage naming.** When no `--stage` is passed, alchemy resolves `--stage` ‚Üí `$STAGE` ‚Üí **`dev_${USER}`** ‚Üí `dev_unknown` (`alchemy@2.0.0-beta.59 ‚Äî src/Cli/commands/_shared.ts`). So a bare `alchemy dev` or `alchemy deploy` on a laptop lands in a personal `dev_<user>` stage. `Test.make` defaults to stage `"test"` (`src/Test/Core.ts`). phoenix's stage vocabulary: `prod` (CI, push to main), `pr-<n>` (CI previews, torn down on PR close), `it-*` (run-unique integration stages, torn down in `afterAll`), `dev_<user>` (personal dev). Stage names must match `[a-z0-9][-_a-z0-9]*`.

**Physical names carry a random per-instance suffix.** An auto-named resource's physical name is `${stack}-${id}-${stage}-<suffix>`, where the suffix is 16 base32 chars of a freshly generated random 128-bit `InstanceId` (`alchemy@2.0.0-beta.59 ‚Äî src/PhysicalName.ts`, `src/InstanceId.ts`) ‚Äî e.g. `phoenix-phoenix-db-dev-usirin-4luzwrj6sdbjakbw`. That `InstanceId` is persisted **only in the state store**: the state entry is the one link between the logical resource and the physical database.

**Adopt-vs-create.** On a deploy with no prior state for a resource, the engine runs an adoption probe: it calls the provider's `read`; a resource found and owned is silently re-adopted (no flag), one found but foreign fails `OwnedBySomeoneElse` unless `--adopt` takes it over, and one not found is created (`alchemy@2.0.0-beta.59 ‚Äî src/AdoptPolicy.ts`, `src/Plan.ts`). Alchemy's docs call recovery-into-a-fresh-state-store "the default" ‚Äî **but that promise only holds for resources whose physical name is recomputable.** The probe runs with a *freshly generated* `InstanceId` (`src/Plan.ts`, the `adoptInstanceId` branch), and D1's `read` falls back to a name lookup built from exactly that (`src/Cloudflare/D1/Database.ts` ‚Äî `createDatabaseName(id, olds?.name)`, which is `name ?? createPhysicalName({id})`). With an explicit stable `name` prop the lookup matches and the DB is re-adopted; with an auto-generated name the fresh random suffix guarantees a miss, so the engine **creates a brand-new database and the old one is orphaned** ‚Äî invisible to alchemy forever, because state was the only link and `destroy` is state-driven.

**Why losing local state orphaned cloud resources ‚Äî and how the dev loop was fixed (#2361).** Put the pieces together for the dev loop as it *was*: `alchemy dev` creates a *real* D1 in `dev_<user>` (no local D1 provider ‚Äî see the dev-loop note above), recorded only in `<cwd>/.alchemy/state/` ‚Äî a gitignored, per-checkout, per-worktree directory. When that D1 was **auto-named** (a random suffix), deleting `.alchemy`, cloning fresh, or spinning up a new worktree left the next `alchemy dev` unable to re-find it: it minted a new one, seconds apart if you retried. That is precisely the ~20-strong `phoenix-phoenix-db-dev-usirin-*` / `phoenix-phoenix-db-test-*` orphan cohort of #2340.

The **minting side is now closed for the dev loop**: `PhoenixDb` (`worker/db/resources.ts`) carries a stable, stage-derived explicit `name` **on the local-state dev path only** (`devDatabaseName(process.env)` in `worker/env.ts`, keyed off the same `resolveStateMode`/`ALCHEMY_EXEC_OPTIONS.stage` seam that selects `localState()`), so `${stack}-${id}-${stage}` (e.g. `phoenix-phoenix-db-dev-usirin`) is recomputable and the adoption probe re-links the same D1 across worktrees / deleted `.alchemy/` / new machines (#2361). The name is deliberately `undefined` on every **hosted-state** path (production, a real `alchemy deploy`, the `Test.make` integration harness), so their D1s stay auto-named ‚Äî naming an *already-tracked* D1 is a `diff ‚Üí replace` (`src/Cloudflare/D1/Database.ts`, `DatabaseProvider.diff`), which would replace the production database with an empty one. So the `it-*`/`pr-*` and any pre-#2361 dev orphans are still swept, but no *new* dev-loop orphans are minted. Behavior change (accepted, #2361): concurrent worktrees in one personal stage now share that stage's single dev D1 ‚Äî stage is the isolation unit (ADR [0057](../.decisions/0057-multi-app-multi-worker-repo.md)); use `--stage` for a worktree that needs its own.

**The discipline:**

- **Destroy before you reset.** `alchemy destroy --stage <name>` *before* deleting `.alchemy/`, discarding a worktree that ran `alchemy dev`, or otherwise abandoning a stage. `destroy` is deploy-with-desired-state-zeroed, driven by the state store ‚Äî once the state is gone there is nothing left to destroy *from*.
- **Local state is for genuinely disposable, personal stages only** ‚Äî and even then only under the destroy-before-reset rule. Anything shared, long-lived, or CI-driven uses `Cloudflare.state()` (phoenix already does: deploys and the integration harness are hosted-state; only `alchemy dev` is local).
- **Never rely on adopt-on-recreate for an auto-named resource.** It cannot work (random suffix, above). A resource that must survive state loss needs a stable explicit `name` ‚Äî at the cost of hand-managing per-stage uniqueness. This is exactly why `PhoenixDb` carries a stage-derived `name` **on the local-state dev path** (`devDatabaseName`, #2361) and stays auto-named on hosted state (naming a tracked D1 is a destructive `replace`).
- **Inspect before you clear.** `alchemy state` lists stacks ‚Üí stages ‚Üí resources (`--local` reads `.alchemy/state`); `alchemy state clear` deletes state entries *without touching cloud resources* ‚Äî i.e. it manufactures the orphan condition deliberately, so destroy first unless re-adoption is actually possible.
- **Already-orphaned resources are recovered account-side only:** the state store no longer knows them, so it's the orphan-sweep (`packages/orphan-sweep`), `alchemy nuke` (state-blind, account-level provider enumeration ‚Äî a blunt, whole-account instrument), or a manual API delete.
