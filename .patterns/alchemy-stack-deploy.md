# The stack & deploy

How phoenix is declared and shipped. The short answer: `alchemy.run.ts` is an Effect program — an `Alchemy.Stack` — that declares the resources (the D1 database, the worker) and returns the stack's outputs. `alchemy deploy` runs it against the Cloudflare API; `alchemy dev` runs it locally. There is no `wrangler.jsonc`.

## The stack

```ts
// apps/web/alchemy.run.ts
import * as Alchemy from "alchemy";
import * as Cloudflare from "alchemy/Cloudflare";
import * as Effect from "effect/Effect";
import {resolveStateMode} from "./worker/env.ts";
import PhoenixLive, {Phoenix} from "./worker/index.ts";

export default Alchemy.Stack(
  "phoenix",
  {
    providers: Cloudflare.providers(),
    state:
      resolveStateMode(process.env) === "cloudflare" ? Cloudflare.state() : Alchemy.localState(),
  },
  Effect.gen(function* () {
    const worker = yield* Phoenix;
    return {url: worker.url};
  }).pipe(Effect.provide(PhoenixLive)),
);
```

- **`providers`** — the resource implementations the stack may use. `Cloudflare.providers()` covers Workers/D1/DOs/assets — phoenix's full surface. (There is no `Drizzle.providers()` in the stack: migration SQL is generated by `drizzle-kit` out-of-band; alchemy only applies it via the D1 resource's `migrationsDir`. See [alchemy-drizzle-d1.md](./alchemy-drizzle-d1.md).)
- **`state`** — where alchemy stores the deployed-resource state it diffs against. Two choices, and they decide whether dev is offline: `Cloudflare.state()` is a **Cloudflare-hosted** store (reproducible across machines/CI, but its Layer needs credentials + network on *every* run, including `alchemy dev`); `Alchemy.localState()` is a **file-based** store (Layer needs only `FileSystem`/`Path` — no creds, no network). phoenix picks the store from a dev-vs-deploy signal (`resolveStateMode` in `worker/env.ts`), **not** from `CI` — `CI` is set for both the deploy workflow and the integration-test job and can't distinguish them. Only `alchemy dev` and the Vitest harness resolve to `localState()` (offline); a real `alchemy deploy` always uses the shared Cloudflare store. With `localState()`, `alchemy dev` boots with no auth prompt and zero external connections; `Cloudflare.state()` requires `alchemy login`.
- **body** — yields the resources to deploy and returns the stack outputs (the deployed URL, etc.).

Yielding `Phoenix` deploys the worker; the worker's own init phase (its `bind()` calls and `DurableObjectNamespace` declarations) tells alchemy which bindings, DOs, and migrations to send. The stack doesn't re-declare them.

## Resources shared between stack and worker

Resource *declarations* the worker binds — the D1 database — live in a module both import, so there's one definition:

```ts
// apps/web/worker/db/resources.ts
export const PhoenixDb = Cloudflare.D1Database("phoenix_db", {
  migrationsDir: "./worker/db/drizzle/migrations",
  migrationsTable: "drizzle_migrations",
});
```

Migration SQL under `migrationsDir` is generated out-of-band by `drizzle-kit` (against `worker/db/drizzle.config.ts`) and committed; there is no `Drizzle.Schema` resource. `migrationsTable` matches drizzle-kit's bookkeeping table so the applied-set is shared between generator and applier. See [alchemy-drizzle-d1.md](./alchemy-drizzle-d1.md).

> **`migrationsDir` ≠ adopting a DB — and phoenix had nothing to adopt.** `migrationsDir` only tells alchemy where the migration SQL lives, not which physical DB binds. `Cloudflare.D1Database` can adopt an existing DB **by name** (it lists and reuses a DB matching the name; no `adopt` flag needed) and `migrationsDir` migrations are idempotent (already-applied skipped, new ones applied — `migrationsTable: "drizzle_migrations"` for Drizzle compatibility). phoenix's prod D1 held **no irreplaceable data** — [ADR 0009](../.decisions/0009-d1-direct-defer-dos-and-workflows.md) is explicit ("D1 is wiped and re-seeded; no live data"), and the seeders are dev-only-gated so prod can't even be populated. So the deploy uses a **fresh D1 and re-seeds** via the import scripts rather than adopting — see the cutover note under [Dev & deploy](#dev--deploy).

The worker `bind()`s `PhoenixDb` ([alchemy-bindings.md](./alchemy-bindings.md)); the stack ensures the DB exists and its committed migrations have been applied before the worker deploys. The Durable Objects (`ConnectionDO`, `TopicDO`) are the `DurableObjectNamespace` classes themselves — declaring them and binding them in the worker is enough; alchemy derives the migrations. Under the hood alchemy tracks DO migrations via Cloudflare **worker tags** (`alchemy:do:{id}:{class}`, `alchemy:migration-tag:{version}`), **defaults new DO classes to `new_sqlite_classes`**, and computes `deletedClasses` by diffing the old-vs-new tag maps.

> **Cutover went fresh on both axes (no data to preserve).** phoenix is effectively greenfield: prod ran but the D1 held no irreplaceable data ([ADR 0009](../.decisions/0009-d1-direct-defer-dos-and-workflows.md)). So:
> - **Fresh D1** — `Cloudflare.D1Database` creates a new database and applies migrations. (Adoption-by-name is available if ever needed, but unnecessary here.) The old `import-sozluk`/`import-pano` seeders that re-populated content were deleted; future seeding is a direct-D1 script.
> - **Fresh worker script name** — deploying under a new name avoids two hazards of reusing the wrangler-managed `phoenix` script: (1) alchemy brands a script with no `alchemy:*` tags as **`Unowned`** and refuses to deploy without an explicit `--adopt`; (2) the DO migration-tag precondition won't match wrangler's existing tags (alchemy *does* have a self-healing recovery that parses the expected tag from the error and retries, but a fresh name skips the dance entirely).
> The old wrangler-managed `phoenix` worker + `phoenix_db` were deleted once the alchemy worker was verified. Reusing the same script name is technically supported (`--adopt` + the tag-recovery path) but strictly higher-risk for zero benefit, since there's no data to keep.

## What `wrangler.jsonc` keys become

| `wrangler.jsonc` | alchemy |
|---|---|
| `name`, `main`, `compatibility_*` | `Cloudflare.Worker` id + props ([alchemy-worker.md](./alchemy-worker.md)) |
| `assets` | the `assets` prop on the worker |
| `d1_databases` | `Cloudflare.D1Database(...)` resource |
| `durable_objects.bindings` | `Cloudflare.DurableObjectNamespace` declarations |
| `migrations` (DO classes) | derived from the DO declarations |
| `d1 migrations_dir` | `D1Database({migrationsDir, migrationsTable})` ([alchemy-drizzle-d1.md](./alchemy-drizzle-d1.md)) — SQL generated by `drizzle-kit` out-of-band |
| `vars` | resource props / `Layer.succeed` config |

## Dev & deploy

```bash
pnpm build        # vite build → dist/client (the SPA assets the worker serves)
alchemy deploy    # bundle the worker, upload dist/client, push to the Cloudflare API
alchemy deploy --stage prod
```

The SPA is built by Vite as a normal build step (`dist/client`), then uploaded via the worker's `assets` prop — `alchemy deploy` does not drive Vite for phoenix's single-worker shape ([alchemy-worker.md](./alchemy-worker.md) explains why it's `Cloudflare.Worker` + `assets`, not `Cloudflare.Vite`). Drop `@cloudflare/vite-plugin` from `vite.config.ts` (alchemy is incompatible with it); keep `react()` and the `fate()` codegen plugin, which reads the server's `Entity<>` types regardless of deploy path (see [fate-server-wiring.md](./fate-server-wiring.md)).

### Local dev — two processes (the decided model)

phoenix's dev loop is **`vite dev` for the SPA (HMR) + `alchemy dev` for the worker, with Vite proxying the API**. This is a deliberate choice, not a workaround — here's the constraint that forces it:

alchemy has **two non-mixable worker runtime paths**. The Effect-native path (`Cloudflare.Worker` + `bind()` + the Effect DO model — what phoenix uses) runs under `alchemy dev`'s own local workerd runtime: it watch-rebuilds the *backend* on change but serves `dist/client` **statically, with no client HMR**. Integrated HMR exists only on the *other* path — `Cloudflare.Vite` driving `@distilled.cloud/cloudflare-vite-plugin` (alchemy's own fork) — but that path's worker entry is a plain `export default {fetch}`, so it **can't host phoenix's Effect-native worker**. You can't get `bind()`/Effect-DOs *and* integrated HMR on one worker.

> **Don't try to "fix" this by adding a Vite plugin to the Effect-native worker.** The reason there's no single-process HMR isn't a missing config — it's the two-path split above. `@distilled.cloud/cloudflare-vite-plugin` only runs the `Cloudflare.Vite` (plain-handler) path. phoenix chose the Effect-native worker and takes the two-process dev loop as the price.

So `alchemy dev` runs the one worker (live D1/DO bindings, backend watch-rebuild), and Vite serves the SPA with HMR and proxies the API to it. `alchemy dev` serves the worker **vhost-routed** at `http://<worker-name>.localhost:1337` — and since Node can't resolve `*.localhost`, the Vite proxy must target the IP and force the `Host` header (a `target: "http://<name>.localhost:1337"` fails with `ENOTFOUND`):

```ts
// vite.config.ts — dev only; alchemy owns deploy
const worker = {
  target: "http://127.0.0.1:1337",        // alchemy dev's local proxy port
  changeOrigin: false,
  headers: {host: "phoenix.localhost"},   // route to the worker by its vhost name
};
server: {proxy: {"/api": worker, "/fate": worker}}; // /fate/live SSE streams through fine
```

```bash
alchemy dev     # one worker + DOs + D1, vhost at http://phoenix.localhost:1337 (offline with localState; Cloudflare.state needs alchemy login)
pnpm vite dev   # SPA + HMR + the fate() codegen plugin; proxies /api, /fate → the worker
```

> **How the dev loop behaves** (alchemy `2.0.0-beta.44`, effect v4). End-to-end in a browser: `alchemy dev` runs the Effect-native `Cloudflare.Worker`; an SSE stream (`HttpServerResponse.stream`) flows through the Vite proxy live; and **editing a React component does not drop the SSE connection** (the stream runs unbroken across edits — the proxied SSE socket isn't on Vite's HMR module graph). Three gotchas baked into the config above: (1) **state-store choice decides offline** — with `Cloudflare.state()` `alchemy dev` needs `alchemy login` + network every run, but with `Alchemy.localState()` it boots with no auth and zero external connections (fully offline; see the `state` bullet); (2) the `127.0.0.1` + `Host` proxy shape is mandatory (`*.localhost` is unresolvable in Node); (3) the install peers are `effect@beta` (v4; npm `latest` is v3), `@effect/platform-node`, `@effect/platform-bun`.

It stays **one worker** — the second terminal is just the Vite dev server (not a worker); at deploy `vite build` produces `dist/client` and the single worker serves it. React HMR comes from `vite dev`, backend hot-reload from `alchemy dev`, so the full loop is live. A simpler one-terminal fallback is `alchemy dev` + `vite build --watch` (the static asset server picks up rebuilt `dist/client` on full reload, no HMR).

> **Two honest costs vs the old `@cloudflare/vite-plugin` loop.** (1) Two terminals instead of one command. (2) In dev the SPA is served by `vite dev` and routes to the worker through the Vite proxy, so dev routing is proxy rules rather than the prod `runWorkerFirst`/`assets` precedence — a dev/prod fidelity gap to keep in mind when debugging routing. Both are accepted tradeoffs of the Effect-native single-worker choice, not blockers.

> **Session-gated SSE works through the proxy.** The better-auth session cookie survives the proxy and `/fate/live` validates it via `EventSource(..., {withCredentials:true})` — no proxy or auth config change needed. It works because phoenix's better-auth (1.6.10) emits a dev cookie with **`Secure=false`, no `Domain`, `SameSite=Lax`, `HttpOnly`** (host-only on the Vite origin, stored over `http://localhost`, forwarded through the proxy). Two caveats this surfaced: (a) **don't flip `Secure` on in dev** — setting `NODE_ENV=production`, an `https://` `baseURL`, or `advanced.useSecureCookies:true` makes the cookie un-storable on `http://localhost` and breaks the dev loop; (b) the worker sees `Host: 127.0.0.1:<port>` (the proxy socket), not `<name>.localhost`, so any better-auth `baseURL`/`trustedOrigins` derived from the inbound Host header would read the proxy address in dev — set those explicitly rather than inferring from Host.

### Tests run on `alchemy/Test`, not `vitest-pool-workers`

The test harness is **`alchemy/Test/Vitest`** because **`@cloudflare/vitest-pool-workers` cannot run an alchemy worker.** The pool loads the worker module inside workerd, but the alchemy `Cloudflare.Worker` class is a deploy-time *Resource* that transitively imports alchemy's bundler (`rolldown`'s native `.node` binding + Node-only modules) — workerd can't load it (the real workerd entry is a virtual module alchemy generates at bundle time). It's not a config problem (the pool *can* run without `wrangler.jsonc`, via inline miniflare options) — the alchemy worker simply isn't loadable that way.

The harness (offline) deploys the stack to a local workerd (`dev: true` + `Alchemy.localState()`) in `beforeAll`, then asserts over HTTP against `stack.url`. The integration suite was ported as: `SELF.fetch(...)` → `HttpClient` against `stack.url`; direct `env.PHOENIX_DB.prepare(...)` read-back assertions → black-box HTTP via a worker route; and the two DO-internal tests (`runInDurableObject`/alarm, esp. `fate-live.test.ts`) → black-box SSE/timing.

> **Stages give isolated environments per branch/PR.** `--stage <name>` deploys an independent copy of the stack (its own DOs, its own D1). This is how preview deploys work without a second config file — the stage name is threaded into resource names. `alchemy.run.ts` can branch on `stage` (e.g. reference a shared staging D1 for `pr-*` stages) the way the alchemy Neon examples do.

> **Tooling note.** alchemy's own examples run under `bun`; phoenix runs under `pnpm` + node. alchemy is plain TypeScript, so `pnpm alchemy deploy` / a node-run entry works. The `package.json` scripts, `turbo.json`, and CI run the alchemy toolchain; `wrangler` is gone.

> **Version pinning.** The workspace catalog pins exact versions: `alchemy 2.0.0-beta.44` and `effect 4.0.0-beta.70` (with the matching `@effect/platform-*` / `@effect/vitest` betas). These are exact pins, not floating `alchemy@next` / `effect@beta` tags. If they ever bump, **re-verify the DO migration behavior on the new versions** — the tag-diffing/`new_sqlite_classes` logic above was confirmed at these beta versions.
